VirtualBox Network Configuration

Configuring the Host-Only adapter so the client and domain controller can communicate.

⇄
Host-Only Adapter Explained
A Host-Only adapter creates a private network between the host machine and virtual machines. Both DC1 and the client must use the same Host-Only adapter to communicate with each other. This network is isolated from the physical network, making it ideal for lab environments.
DC1
192.168.56.120
VirtualBox Host-Only Network
Win10-Client
192.168.56.121
⚙
Configuration Steps
  1. Shut down the Win10-Client VM if it is running
  2. Open Settings for the Win10-Client VM
  3. Navigate to the Network tab
  4. Set Adapter 1 to Host-Only Adapter
  5. Select the same adapter used by DC1: VirtualBox Host-Only Ethernet Adapter
  6. Click OK to save the changes
ⓘ
Why Host-Only? This ensures both VMs are on the same private network segment. Without this, the client cannot reach the domain controller, and domain joining will fail.
05

Client IP Configuration

Assigning a static IP address and pointing DNS to the domain controller.

⚙
Static IP Assignment
After booting Windows 10, configure the network adapter with a static IP address:
  1. Press Windows+R to open the Run dialog
  2. Type ncpa.cpl and press Enter
  3. Right-click the Ethernet adapter and select Properties
  4. Select Internet Protocol Version 4 (TCP/IPv4)
  5. Click Properties
FieldValue
IP Address192.168.56.121
Subnet Mask255.255.255.0
Default GatewayLeave Blank
Preferred DNS Server192.168.56.120
⚠
CRITICAL: DNS MUST point to the Domain Controller (192.168.56.120). Without correct DNS configuration, domain joining will fail. The client uses DNS to locate the domain controller for authentication and policy updates.
ⓘ
Why Leave Gateway Blank? In a lab environment with a Host-Only network, there is no router. The gateway is unnecessary because all systems are on the same subnet. In production, you would configure the correct gateway address.

DNS Configuration Fundamentals

Understanding why DNS is critical for Active Directory to function correctly.

ⓘ
DNS and Active Directory
Active Directory Domain Services relies on DNS for the following critical functions:
  • ▸ Locating Domain Controllers — Clients query DNS to find DC1
  • ▸ Domain Name Resolution — Resolves ignite.local to an IP address
  • ▸ Service Location (SRV Records) — Identifies LDAP, Kerberos, and other services
  • ▸ Authentication Routing — Directs authentication requests to the correct server
Win10-Client
DNS Query: "Where is ignite.local?"
DC1 · DNS Server
Response: "ignite.local = 192.168.56.120"
Win10-Client initiates Domain Join
ⓘ
Remember: Without DNS pointing to DC1, the client cannot resolve the domain name ignite.local. Domain Join, authentication, and Group Policy updates will all fail.

Connectivity Verification

Verifying that the client can communicate with the domain controller before attempting a domain join.

⚙
Verification Tests
Open Command Prompt and run the following tests to verify connectivity:
Command Test 1 — Ping DC1
C:\Users\student> ping 192.168.56.120
ⓘ
Ping Test: Verifies basic network connectivity between the client and DC1. If you receive replies, Layer 3 (IP) connectivity is working.
Command Test 2 — DNS Lookup
C:\Users\student> nslookup ignite.local
ⓘ
DNS Test: Confirms DNS is functioning correctly. The response should show the server as DC1 (192.168.56.120) and resolve ignite.local to the same address. If this works, DNS is properly configured.
⚠

Domain Authentication Verification

Proving that authentication now happens through Active Directory rather than the local machine.

✓
Verification Commands
After reboot, log in using a domain user account (e.g., ignite\yashika). Then verify domain authentication:
Command — whoami
C:\Users\yashika> whoami
ⓘ
whoami shows the current user identity. If the output is ignite\yashika, authentication is now handled by Active Directory, not the local machine.
Command — Logon Server
C:\Users\yashika> echo %logonserver%
ⓘ
%logonserver% reveals which domain controller authenticated the user. The output \\DC1 confirms that DC1 handled the authentication request.
✓
Success Criteria: whoami shows ignite\username and %logonserver% shows \\DC1. Both confirm Active Directory is functioning correctly for client authentication.

Moving Computer Objects into OUs

Organizing computer objects into Organizational Units for targeted policy application.

ⓘ
Why Move to an OU?
By default, computer objects are placed in the Computers container. However, the Computers container is not an Organizational Unit (OU) and cannot have Group Policy Objects linked to it. To apply GPOs to a computer, it must be moved into a custom OU.
✓
OU Structure Example
ignite.local ├── HR │ └── WIN10-CLIENT ├── IT ├── Finance ├── Users └── Computers
⚙
Move Procedure
  1. Open Active Directory Users and Computers on DC1
  2. Navigate to the Computers container
  3. Right-click WIN10-CLIENT
  4. Select Move
  5. Choose the target OU (e.g., HR)
  6. Click OK
ⓘ
Benefit: Once moved to the HR OU, Group Policies linked to that OU will automatically apply to WIN10-CLIENT. This enables targeted policy management based on department or function.

Group Policy Management Console

Opening and navigating the Group Policy Management Console on the Domain Controller.

⚙
Opening GPMC
  1. On DC1, open Server Manager
  2. Click the Tools menu in the top-right corner
  3. Select Group Policy Management
  4. In the console, expand Forest: ignite.local
  5. Expand Domains → ignite.local
ⓘ
Default GPOs Explained
When you open GPMC, you will see these default policies:
GPO NamePurpose
Default Domain PolicyApplied to all domain users and computers. Contains baseline settings like password policy, Kerberos policy, and account lockout.
Default Domain Controllers PolicyApplied only to Domain Controllers. Contains security-sensitive settings for DCs.
ⓘ
Best Practice: Never modify the Default Domain Policy for application-specific settings. Create separate GPOs for each policy purpose and link them to specific OUs. This makes troubleshooting and delegation easier.

Creating a New GPO

Creating a custom Group Policy Object for targeted policy application.

✓
GPO Creation Steps
We will create a GPO linked to the HR Organizational Unit:
  1. In GPMC, locate your target OU (e.g., HR)
  2. Right-click the HR OU
  3. Select Create a GPO in this domain, and Link it here...
  4. Enter the name: HR Security Policy
  5. Click OK
ignite.local ├── HR │ └── HR Security Policy ├── IT ├── Finance └── ...
ⓘ
What happened? A new Group Policy Object named "HR Security Policy" was created and linked to the HR OU. Any policy settings configured within this GPO will apply to all users and computers in the HR OU.
15

Linking GPO to Organizational Units

Understanding how GPO linking determines which users and computers receive which policies.

⇄
GPO Link Inheritance
GPOs can be linked at three levels in Active Directory, each with a specific scope and precedence order:
Site Level — Affects all sites
▼ (Lowest precedence)
Domain Level — Affects all domain members
▼ (Medium precedence)
OU Level — Affects only that OU
▼ (Highest precedence)
ⓘ
Inheritance Rule: By default, child OUs inherit GPOs from parent containers. You can block inheritance or enforce a GPO link to override this behavior. In our lab, linking to the HR OU means only objects within HR receive these policies.

Password Policy Configuration

Configuring domain-wide password policies using the Default Domain Policy.

ⓘ
Password Policy Best Practices
Password policies are typically configured at the domain level (Default Domain Policy) because they apply to all domain users. This ensures consistent security across the organization.
⚠
Note: Password policies are a domain-wide setting. They should be configured in the Default Domain Policy, not in custom GPOs linked to OUs. Password settings in OUs will not override domain-level password policy.
⚙
Configuring Password Policy
  1. Right-click Default Domain Policy and select Edit
  2. Navigate to: Computer Configuration → Policies → Windows Settings → Security Settings → Account Policies → Password Policy
Policy SettingConfigured ValueDescription
Minimum Password Length8 charactersPasswords must be at least 8 characters long
Password must meet complexity requirementsEnabledRequires uppercase, lowercase, digits, and special characters
Account Lockout Threshold5 invalid attemptsAccount locks after 5 failed password attempts
ⓘ
Teaching Point: After configuring these settings, every domain user will be required to follow these password rules. Users cannot set weak passwords like "123" or "password". The Domain Controller enforces these policies centrally.
17

Control Panel Restriction Policy

Restricting access to Control Panel and PC settings using Group Policy.

⚠
Why Restrict Control Panel?
In enterprise environments, standard users should not be able to change system settings through Control Panel. Restricting access prevents users from modifying network settings, uninstalling applications, or changing security configurations.
⚙
Configuration Steps
  1. Right-click HR Security Policy and select Edit
  2. Navigate to: User Configuration → Policies → Administrative Templates → Control Panel
  3. Open Prohibit access to Control Panel and PC settings
  4. Select Enabled
  5. Click Apply and OK
✓
Result: When this policy is applied, if a user tries to open Control Panel (by running control or navigating through Start Menu), they will receive an error: "This operation has been cancelled due to restrictions in effect on this computer."
ⓘ
Teaching Point: The administrator never touched the client machine directly. The restriction came from the Domain Controller through Group Policy. This is the power of centralized management.

19

USB Restriction Policy

Blocking removable storage devices using Group Policy to prevent data exfiltration and malware.

⚠
Why Restrict USB Devices?
USB drives are one of the most common vectors for data theft and malware introduction. In enterprise environments, organizations often restrict or block removable storage to protect sensitive data and maintain security compliance.
⚙
USB Restriction Configuration
  1. Edit HR Security Policy
  2. Navigate to: Computer Configuration → Policies → Administrative Templates → System → Removable Storage Access
  3. Open All Removable Storage classes: Deny all access
  4. Select Enabled
  5. Click Apply and OK
⚠
Impact: This policy blocks all removable storage devices including USB drives, external hard drives, memory cards, and portable media players. Users will not be able to access or use these devices on their workstations.
✓
Enterprise Relevance: USB restrictions are commonly required for compliance with standards like ISO 27001, NIST, and GDPR. Group Policy makes it possible to enforce this across the entire organization from a single console.

Force Policy Update

Applying Group Policy changes immediately rather than waiting for the automatic refresh cycle.

ⓘ
Understanding Policy Refresh
By default, Group Policy refreshes automatically every 90 minutes (with a random offset of 0-30 minutes). However, administrators typically don't wait — they force an immediate update.
⚙
Force Update Procedure
  1. Login to the Windows 10 Client using a domain user: ignite\yashika
  2. Open Command Prompt (Run as Administrator) or PowerShell (Run as Administrator)
Command — gpupdate /force
C:\Windows\system32> gpupdate /force
ⓘ
What happens: The client contacts DC1 and asks "Do you have any new policies for me?" DC1 responds with the latest GPO settings, and the client applies them immediately. If prompted to log off or restart, press Y.
21

GPO Verification

Verifying that Group Policies have been successfully applied to the client machine.

⚠
A Common Mistake
Many beginners run gpupdate successfully but never verify whether policies actually applied. Real administrators ALWAYS verify.
✓
Verification Commands
Command — gpresult /r
C:\Users\yashika> gpresult /r
ⓘ
gpresult /r shows a summary of which GPOs were applied to the current user and computer. Look for HR Security Policy in the "Applied Group Policy Objects" section. If you see it, the GPO reached the machine successfully.
⚙
Advanced Verification
Command — gpresult /h
C:\Users\yashika> gpresult /h report.html
ⓘ
Better Method: Running gpresult /h report.html generates an HTML report that you can open in a browser. The report visually displays user settings, computer settings, applied GPOs, denied GPOs, and security filtering details. This is how enterprise administrators troubleshoot GPO issues.
